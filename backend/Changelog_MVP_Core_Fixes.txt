FIX 3: Gini fairness score — implemented from scratch
──────────────────────────────────────────────────────────────────────────────
FILE: app/services/fairness.py (NEW FILE)

PROBLEM:
  The fairness score was described in detail in About App.txt but had zero
  code. This was one of your core deliverables.

WHAT WAS ADDED:
  - calculate_gini_coefficient(workloads: List[int]) → float
  - Formula: G = (2 * Σ(i * y_i)) / (n * Σ(y_i)) - (n+1) / n
  - Edge cases: empty list → 0.0, single member → 0.0, all zeros → 0.0
  - Result clamped to [0.0, 1.0], rounded to 4 decimal places

HOW IT WORKS:
  from app.services.fairness import calculate_gini_coefficient

  # Equal workload → low score
  calculate_gini_coefficient([10, 10, 10])  # → 0.0

  # Unequal workload → higher score
  calculate_gini_coefficient([10, 20, 30])  # → 0.2222

  # Very unequal → close to 1
  calculate_gini_coefficient([0, 0, 100])   # → 0.6667

HOW TO INTERPRET:
  0.0 - 0.15  →  Excellent (very fair distribution)
  0.15 - 0.30 →  Good (minor imbalance, likely acceptable)
  0.30 - 0.50 →  Warning (noticeable imbalance)
  0.50+       →  Bad (significant unfairness, needs attention)

FIX 4: Progress tracking — implemented from scratch
──────────────────────────────────────────────────────────────────────────────
FILE: app/services/progress.py (NEW FILE)

PROBLEM:
  Progress calculation was described in docs but had no code. Without it,
  there's no way to show a "45% complete" indicator on the frontend.

WHAT WAS ADDED:
  - calculate_progress(milestones: List[dict]) → ProgressResponse
  - Uses EFFORT POINTS, not task count. A milestone worth 50 points
    counts 5x more than one worth 10.
  - Per-milestone progress + overall project progress.

HOW IT WORKS:
  Milestones have 3 statuses: "pending", "active", "completed".
  
  - "completed" → full effort_points added to completed total
  - "active"    → fraction = (done task complexity) / effort_points
  - "pending"   → 0% progress
  
  Overall = (total completed effort / total project effort) × 100

  from app.services.progress import calculate_progress
  result = calculate_progress([
      {"title": "Planning", "effort_points": 10, "status": "completed", "tasks": [...]},
      {"title": "Frontend", "effort_points": 20, "status": "active", "tasks": [
          {"complexity": 5, "status": "done"},
          {"complexity": 15, "status": "todo"}
      ]},
      {"title": "Testing", "effort_points": 15, "status": "pending", "tasks": []}
  ])
  # result.overall_percent → 33.3  (10 done + 5/20 of 20 = 12.5, total 45 → 12.5/45)

FIX 6: Workload summary builder — created
──────────────────────────────────────────────────────────────────────────────
FILE: app/services/workload.py (NEW FILE)

PROBLEM:
  generate_tasks_for_milestone() required a pre-formatted string like:
    "Alice: completed 12 points, pending 5 points; Bob: ..."
  But nothing built this string. Every test had to type it manually.

WHAT WAS ADDED:
  - build_workload_summary(team_members, tasks) → str
    Tallies completed vs pending points per member from a list of task dicts.

  - build_workload_summary_empty(team_members) → str
    Generates a "zero workload" string for the first milestone.

HOW IT WORKS:
  from app.services.workload import build_workload_summary

  summary = build_workload_summary(members, [
      {"assigned_to": "Alice", "complexity": 5, "status": "done"},
      {"assigned_to": "Alice", "complexity": 7, "status": "doing"},
      {"assigned_to": "Bob",   "complexity": 3, "status": "todo"},
  ])
  # → "Alice: completed 5 points, pending 7 points; Bob: completed 0 points, pending 3 points"

  When Member 5 integrates the DB, this function just needs to receive
  task rows instead of manual dicts — the output format stays the same.


──────────────────────────────────────────────────────────────────────────────
FIX 7: Task generation — added user message role
──────────────────────────────────────────────────────────────────────────────
FILE: app/services/ai_service.py

PROBLEM:
  generate_tasks_for_milestone() only sent a "system" message to the LLM.
  generate_milestones_only() correctly sent both "system" and "user" messages.
  Many LLMs produce more reliable structured output when both roles are present.

WHAT CHANGED:
  Before:
    messages=[{"role": "system", "content": prompt}]
  
  After:
    messages=[
        {"role": "system", "content": prompt},
        {"role": "user", "content": f"Generate tasks for milestone: {milestone_title}"}
    ]

WHY:
  Consistency with generate_milestones_only() and better reliability
  across different LLM providers (Groq, DeepSeek).
